# Team Eisenhower Matrix Design System

## 1. 목적

- 이 문서는 `Team Eisenhower Matrix`의 화면 전반을 하나의 제품처럼 보이게 만들기 위한 공통 디자인 기준이다.
- 레퍼런스는 `Miro`의 장점인 `넓은 캔버스`, `보드 중심 레이아웃`, `카드 기반 협업`, `강한 스캔성`을 참고한다.
- 단, `Miro`를 그대로 복제하지 않는다. 이 제품은 "주간 우선순위 정리"가 핵심이므로, 보드형 협업 UI를 더 단순하고 빠르게 사용하는 방향으로 재해석한다.

## 2. 제품 인상

- `board-first`
- `collaborative`
- `fast-scan`
- `light but confident`
- `structured color`
- `tactile cards`

## 3. 핵심 원칙

### 3.1 한 화면에서 상황을 읽게 한다

- 사용자는 화면 진입 후 3초 안에 `팀`, `주차`, `해야 할 일`, `우선순위 상태`를 읽을 수 있어야 한다.
- 중요한 정보는 상단 헤더와 카드 제목에 집중시키고, 설명은 보조 계층으로 내린다.

### 3.2 카드가 인터랙션의 중심이다

- 이 제품의 핵심 단위는 페이지가 아니라 `Task Card`다.
- 생성, 이동, 수정, 상태 확인은 모두 카드 기준으로 설계한다.
- 카드 자체가 가볍고 잘 보이며, 드래그 가능한 오브젝트처럼 느껴져야 한다.

### 3.3 색은 장식이 아니라 구조다

- 브랜드 컬러는 전역 CTA와 포커스에만 사용한다.
- 사분면 색은 상태 구분과 시선 유도에만 사용한다.
- 한 화면에서 의미 없는 포인트 컬러를 여러 개 쓰지 않는다.

### 3.4 캔버스는 밝고 넓게 유지한다

- 기본 인상은 어두운 대시보드가 아니라 `밝은 협업 보드`다.
- 배경은 비어 보이지 않을 정도의 텍스처만 주고, 실제 정보는 흰색 또는 옅은 표면 위에서 읽히게 한다.

### 3.5 둥근 표면과 얕은 그림자로 물성을 만든다

- 현재 코드처럼 모든 반경을 `0`으로 두면 Miro 레퍼런스와 거리가 멀다.
- 앞으로의 공통 기준은 `중간 정도의 둥근 반경 + 얕은 그림자 + 선명한 경계선`이다.

### 4.3 Typography

기본 스택:

```text
Pretendard, "Noto Sans KR", "Apple SD Gothic Neo", sans-serif
```

현재 구현은 `Noto Sans KR` 중심이므로, 즉시 폰트를 교체하지 않아도 된다. 다만 이후 고도화 시 위 스택으로 통일한다.

| 스타일 | 크기/행간 | 굵기 | 용도 |
| --- | --- | --- | --- |
| Display | `48/56` | `800` | 홈 메인 카피, 주차 헤더 핵심 문구 |
| H1 | `36/44` | `800` | 페이지 최상위 제목 |
| H2 | `28/36` | `700` | 섹션 제목 |
| H3 | `22/30` | `700` | 패널 제목 |
| Title | `18/26` | `700` | 카드/폼 제목 |
| Body | `16/24` | `400` | 기본 본문 |
| Body Strong | `16/24` | `600` | 중요 설명, 정보값 |
| Caption | `13/18` | `500` | 메타 정보, 보조 문구 |
| Button | `14/20` | `700` | 버튼 레이블 |

타이포 규칙:

- 타이틀은 짧고 강하게, 설명은 길어도 2문장 이내로 유지한다.
- 한 화면에서 `Display` 계열은 1개만 허용한다.
- 카드 제목은 최대 2줄, 설명은 최대 2줄로 제한한다.

## 5. Layout System

### 5.1 전역 구조

- 앱은 `밝은 보드 위에 패널과 카드가 올라간 구조`로 통일한다.
- 페이지 상단에는 반드시 `페이지 역할을 설명하는 헤더 패널`이 먼저 나온다.
- 주요 액션은 각 화면에서 시선이 먼저 가는 우상단 또는 카드 하단 끝에 놓는다.

### 5.2 Home

- 홈은 단순 랜딩이 아니라 `팀 생성 + 최근 팀 진입 + 제품 컨셉 이해`를 동시에 처리하는 화면이다.
- 좌측은 가치 제안과 생성 액션, 우측은 내 정보와 소속 팀 목록으로 구성한다.
- 과도한 마케팅 문구보다 `바로 시작할 수 있는 제품성`이 우선이다.

### 5.3 Team Board

- 팀 화면은 이 제품의 핵심이므로 가장 `Miro-like`해야 한다.
- 좌측 컬럼은 `빠른 입력 + inbox`, 우측 대영역은 `2x2 matrix`로 유지한다.
- 상단 헤더는 팀/주차/협업 상태/핵심 액션을 모두 담는 `보드 컨트롤 바` 역할을 해야 한다.
- 매트릭스는 넓어 보여야 하며, 카드 이동 가능성을 시각적으로 즉시 이해할 수 있어야 한다.

### 5.4 My Page

- 마이페이지는 설정 화면이므로 보드보다 밀도를 낮추고, 정보 우선순위를 단순하게 유지한다.
- 설정 패널은 내용 단위로 명확히 분리하고, 동일한 패널 구조를 반복 사용한다.

## 6. Component Rules

### 6.1 Header Panel

- 큰 제목, 보조 설명, 상태 칩, 핵심 액션 버튼을 담는 상단 대표 패널이다.
- 배경은 단색보다 `아주 약한 그라데이션`을 권장한다.
- 패딩은 `24~40px`, 반경은 `4px`, 테두리는 `border.soft` 기준으로 한다.

### 6.2 Section Panel

- 홈 우측 정보 블록, 마이페이지 설정 블록, 매트릭스 래퍼에 사용하는 공통 표면이다.
- 배경은 `rgba(255,255,255,0.82~0.92)` 범위의 밝은 표면을 사용한다.
- 섹션 제목과 설명은 항상 상단에 고정한다.

### 6.3 Task Zone

- `TaskZone`은 카드의 부모가 아니라 `드롭 가능한 의미 영역`이다.
- 각 존은 제목, 설명, 카드 리스트, 빈 상태를 반드시 포함한다.
- 카드가 없을 때도 사용자가 무엇을 해야 하는지 알 수 있게 `empty state` 문구를 넣는다.
- 드롭 hover 상태에서는 `border`, `shadow`, `translateY(-2px)` 정도까지만 반응한다.

### 6.4 Task Card

- 카드의 최우선 정보는 `제목`이다.
- 보조 메타는 `담당자`, `마감일`, `설명` 순으로 배치한다.
- 카드 좌측에는 사분면 accent 보더를 둔다.
- 카드 배경은 항상 밝게 유지한다.
- 드래그 핸들은 카드 우상단에 붙이고, 클릭 영역과 충돌하지 않게 분리한다.
- hover 시 카드 전체가 미세하게 떠야 하고, drag 시 명확하게 분리되어 보여야 한다.

### 6.5 Composer

- 빠른 입력 폼은 "작성을 방해하지 않는 짧은 액션 블록"이어야 한다.
- 필드는 최소 개수만 먼저 노출한다.
- 입력 후 상세 편집은 Drawer에서 처리하는 현재 구조를 유지한다.

### 6.6 Drawer

- 상세 수정은 페이지 전환보다 `하단 시트`가 적합하다.
- Drawer는 `sheet` 컴포넌트처럼 보여야 하며 상단 반경을 크게 준다.
- 제목, 설명, 필드, 저장 액션 순으로 단순하게 배치한다.
- 저장 버튼은 오른쪽 끝 기본 CTA로 유지한다.

### 6.7 Buttons

버튼 우선순위:

1. `Contained / Brand`: 저장, 생성, 초대 링크 복사
2. `Outlined`: 보조 이동, 필터, 뒤로 가기
3. `Text`: 낮은 우선순위 보조 액션

버튼 규칙:

- 기본 높이는 `44px` 이상.
- 아이콘은 가능하면 `Rounded` 계열을 사용한다.
- 한 영역에 primary CTA는 1개만 둔다.

### 6.8 Chips

- 칩은 상태나 맥락을 요약하는 용도에만 사용한다.
- 정보 칩은 2~4개 이내로 제한한다.
- 텍스트 길이가 길어지면 버튼으로 바꾸고, 칩을 억지로 늘리지 않는다.

### 6.9 TextField

- 입력창은 `filled`보다 `outlined + soft surface`가 더 적합하다.
- 라벨은 유지하되, placeholder는 실제 입력 예시를 준다.
- 긴 폼도 한 줄 설명으로 끝내고, 장문의 도움말은 숨긴다.

### 6.10 Toast / Alert

- 성공, 에러, 정보 상태를 빠르게 알리고 곧 사라져야 한다.
- 토스트는 하단 또는 우하단에 고정하되, 드래그와 충돌하지 않게 한다.
- 배너형 `Alert`는 페이지 수준 상태 안내에만 사용한다.

## 7. Motion

- Hover: `120ms ~ 160ms`
- Drag lift: `160ms ~ 180ms`
- Drawer open/close: `220ms ~ 260ms`
- 애니메이션은 `transform`, `opacity`, `box-shadow` 중심으로 제한한다.
- 의미 없는 bounce, scale 과다, blur 과다 효과는 사용하지 않는다.

## 8. Iconography

- 아이콘은 현재처럼 `MUI Rounded` 계열을 기본으로 유지한다.
- 같은 영역에서는 선 굵기와 모서리 스타일을 섞지 않는다.
- 아이콘은 장식이 아니라 `행동 힌트`와 `상태 보조` 역할만 수행한다.

## 9. Copy Rules

- 라벨은 짧고 직접적으로 쓴다.
- 버튼 문구는 동사를 포함한다. 예: `저장`, `빠르게 추가`, `초대 링크 복사`
- 설명문은 "무엇을 할 수 있는지"보다 "지금 여기서 무엇을 하면 되는지"를 말한다.
- 시스템 에러 문구는 원인보다 다음 행동을 먼저 제시한다.

## 10. Accessibility

- 텍스트 대비는 최소 `4.5:1`을 맞춘다.
- 색만으로 상태를 구분하지 않는다. 아이콘, 라벨, 위치를 함께 사용한다.
- 클릭/터치 타겟은 최소 `44px` 이상을 유지한다.
- 포커스 링은 `color.brand.500` 기반으로 명확히 보이게 한다.
- 드래그가 어려운 사용자를 위해 클릭 후 상세 편집 흐름도 동일하게 유지한다.

## 11. 현재 코드 기준 적용 원칙

### 11.1 Single Source of Truth

- 디자인 토큰의 단일 진실 소스는 [`src/theme.ts`](./src/theme.ts)여야 한다.
- 페이지와 컴포넌트에 직접 박혀 있는 색상, 보더, 배경 그라데이션은 점진적으로 토큰화한다.

### 11.2 Theme Preset 운영 방식

- 현재의 `sage / sand / ocean` 프리셋은 유지 가능하다.
- 단, 프리셋은 `브랜드 accent와 배경 톤을 바꾸는 skin layer`여야 한다.
- 타이포, spacing, radius, component hierarchy는 프리셋에 따라 바뀌지 않는다.

### 11.3 Quadrant 데이터 구조

- 현재 [`src/lib/types.ts`](./src/lib/types.ts)의 `MATRIX_QUADRANTS`는 의미상 올바르다.
- 다만 장기적으로는 `accent`, `surface`를 하드코딩하지 말고 theme token에서 읽도록 옮기는 편이 좋다.

### 11.4 컴포넌트 공통화 우선순위

1. `TaskCard`
2. `TaskZone`
3. `Header Panel`
4. `Section Panel`
5. `Button / Chip / TextField`

위 5개만 정리돼도 화면 인상의 80%가 통일된다.

## 13. Do / Don't

### Do

- 밝은 보드형 캔버스 위에 카드와 패널을 명확히 올린다.
- 전역 CTA는 `brand blue`로 통일한다.
- 사분면 색은 상태 구분용으로 제한한다.
- 카드와 패널 반경을 일관되게 유지한다.
- 화면마다 같은 헤더 구조와 패널 구조를 반복한다.

### Don't

- 모든 요소를 네모로 자르지 않는다.
- 모든 패널에 다른 그라데이션을 쓰지 않는다.
- 사분면 색과 브랜드 색을 혼용해 우선순위를 흐리지 않는다.
- 설명 문구를 길게 쓰지 않는다.
- hover, drag, drawer 애니메이션을 과장하지 않는다.
